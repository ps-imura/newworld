CHAPTER 06

初めて使ってみる（最小の対話）

この章のゴール
Claude Code を実際に起動して、最初の対話と最初のファイル作成までを体験する。
「AI に頼んでファイルを作る」とはどういうことかを、自分の手でやってみる。

練習用フォルダを作る

まず、練習で使うフォルダを作ります。本書の推奨ルート ~/dev/ の中に練習用フォルダを作ります（理由は第5章・第7章で解説）。

フォルダ作成は、ターミナル.app または Cursor の内蔵ターミナルどちらでも構いません。今は手早く Mac のターミナル.appでやってみましょう： 

mkdir -p ~/dev/hello-claude

mkdir -p の -p は「親フォルダがなければ一緒に作る」という意味です。 ~/dev フォルダもまだ無ければ、同時に作ってくれます。

Cursor で開く → 内蔵ターミナルから Claude Code を起動

第2章で触れたとおり、Poolside の標準は「Cursor 1ウィンドウで完結」スタイルです。 早速この流れに乗りましょう。

  1. Cursor を起動。メニューの「File → Open Folder」から、いま作った ~/dev/hello-claude を選択
  2. Cursor のウィンドウが開いて、左側にフォルダの中身（まだ空）が表示される
  3. 内蔵ターミナルを開く：メニューの「Terminal → New Terminal」、または control + `（バッククォート）
  4. Cursor の画面下半分にターミナルが現れる。プロンプトを見ると、自動で hello-claude フォルダがカレントになっているはず
  5. その内蔵ターミナルで Claude Code を起動： 
claude
cd 不要のメリットを実感してみてください。Cursor で開いているフォルダが自動的にカレントディレクトリになるので、毎回 cd ~/dev/hello-claude と打つ必要がありません。
ターミナル.app を別途使う場合は cd が必要なので、フォルダ取り違いの事故が起きやすい。Cursor 内蔵ターミナルなら、その心配が無くなります。

Claude Code が立ち上がります。画面の見た目は次のようになります（バージョンによって違います）：

一番下に文字を打ち込める欄があります。ここから AI と対話します。

最初のあいさつ

試しに、こう打って Enter を押してみましょう：

こんにちは。今日が初めての利用です。私の今いるフォルダの中身を教えてください。

Claude が ls 相当のことを実行して、答えを返してくれます。 まだ hello-claude フォルダは空なので「何もありません」と返ってくるはずです。

初めてのファイル作成

今度は、AI に直接ファイルを作ってもらいます。

このフォルダに「hello.txt」というファイルを作って、中に「これは私の最初のファイルです」と書いてください。

Claude は「hello.txt を作成してもいいですか？」のように許可を求めてくることがあります。 内容を確認して、問題なければ「はい」「OK」「Yes」などと答えると実行されます。

Claude Code は勝手にファイルを作ったり削除したりしない仕組みになっています。 実行する前に必ず確認を入れてくれるので、暴走を恐れずに頼ってよいです。

確認してみる

ファイルができているか、Finder で見てみましょう。 Claude Code を一度終了させても、Finder で残ったファイルは確認できます。

終了するには Claude Code の画面で： 

/exit

Finder で ~/dev/hello-claude/ を開くと、hello.txt ができているはずです。 ダブルクリックして開いてみると、書いた内容が入っています。

「会話の続きから戻る」やり方

Claude Code を終了したあと、また同じ続きから話したい時の流れ：

Cursor 内蔵ターミナルから（推奨）

  1. Cursor を起動して、~/dev/hello-claude フォルダを開く（前回開いていれば自動で開く）
  2. control + ` で内蔵ターミナルを開く
  3. カレントディレクトリは自動で hello-claude フォルダになっている。cd 不要。下記を実行：
claude --continue

ターミナル.app から（補助）

Cursor を開かずに会話を再開したい時は、ターミナル.app から： 

cd ~/dev/hello-claude
claude --continue

--continue（続ける、の意味）を付けると、そのフォルダで前回した会話の続きから始められます。 ターミナルを閉じても会話が記憶されている、ということです。

「No conversation found to continue」 と出たら、まだそのフォルダで一度も話していない状態です。
最初の1回は claude（オプション無し）で始めて、2回目以降は claude --continue でOKです。

もう少し凝ったお願いをしてみる

再起動後、こう頼んでみましょう：

hello.txt の内容を、もう少し詩的な文章に書き換えてください。

Claude が「こう変えていいですか？」と差分（変更前 → 変更後）を見せてくれます。 気に入った内容なら承認、違うなら「もっと素朴な感じで」のように追加注文ができます。

おい、いまキーボードを見ながら打ってないか？画面と手元を行ったり来たりするタイピングは AI 時代の最大の足かせだ。 AI に指示を出すスピードがそのまま生産性になる時代に、毎回キーを目で探してたら相棒のスピードについていけない。

今のうちから画面だけ見て手元を見ない癖を付けろ。最初は遅くてもいい。1 ヶ月で景色が変わる。 タッチタイピングの練習サイト（e-typing など）を昼休みに 5 分やるだけで全然違うぞ。

ちょっと寄り道：「Markdown（.md）」って何？

AI を使った開発をしていると、.md という拡張子のファイルが頻繁に出てきます。 CLAUDE.md、README.md、SKILL.md など。 これらは「Markdown（マークダウン）」という軽い書式の文書フォーマットです。

どうして Markdown を使うのか

  • 記号がシンプル：見出しは #、リストは -、強調は ** で囲むだけ。HTML の <h1>...</h1> のような長い記述が要らない
  • そのままでも読める：記号入りのテキストでも、人間が読んで意味が分かる。HTML を素のテキストとして読むのと違って、ストレスがない
  • AI が理解しやすい：「これは見出しだな」「これはリストだな」と AI が一発で構造を把握できる。だから Claude Code への指示も CLAUDE.md に Markdown で書く
  • GitHub・各種サービスで表示が綺麗：README.md を GitHub に上げると、自動的に綺麗な見た目で表示される
  • 変換が容易：Markdown → HTML、Markdown → PDF など、後から別の形式に変換するのが簡単

最低限の書き方（これだけ覚えればOK）

# 大見出し（H1）
## 中見出し（H2）
### 小見出し（H3）

これは普通の段落です。

- 箇条書き 1
- 箇条書き 2
- 箇条書き 3

1. 番号付きリスト 1
2. 番号付きリスト 2

**太字** と *斜体* も書けます。

`インラインコード` はバッククォートで囲みます。

```
コードブロックは
バッククォート 3 つで囲む
```

> 引用は > から始める

[リンクテキスト](https://example.com)

![画像の説明](image.png)

これだけで、ほぼ全ての文書を構造的に書けます。HTML より圧倒的に少ない記号数で済みます。

なぜ AI 開発で Markdown が標準なのか

Claude Code・Cursor・GitHub・Notion・Slack など、AI 開発に関わるツールはほとんどが Markdown 表示に対応しています。 つまり 「Markdown で書いておけば、どこに貼っても綺麗に表示される」。 そして AI も Markdown の構造を理解して動くので、指示文や設定ファイルを Markdown で書くと意図が伝わりやすい。

本書を進める中で .md ファイルが出てきたら、「これは構造化された文書だな」と思って中身を見れば、たいてい意味が分かります。 完璧に書ける必要はありませんが、最低限「見出し・リスト・コードブロック」が書けると十分です。

Markdown の書き方を、初心者向けにチートシート（早見表）の形でまとめてください。
よく使うものを上から順に並べ、それぞれに「実際に書いた例」と「画面でどう見えるか」を併記してください。
Cursor で .md ファイルを開くと、画面右上にプレビューボタンが出ます。 記号付きのテキストと、整形された見た目を並べて確認できるので、最初の練習にぴったりです。

脇道課題：Markdown を Notion に貼って「人間向けに展開」してみる

Markdown はそのままだと記号入りのテキストです。 AI や開発者には読みやすいですが、社内の他メンバーやクライアントに見せる時は「綺麗に整形された見た目」で渡したいですよね。

Markdown は主要なドキュメントツールに貼り付けるだけで自動整形されるのが大きな強みです。 ここでは Poolside でも使う Notion（ノーション） に貼って、変換のされ方を実体験してみましょう。

Notion とは（軽く）

  • Notion = メモ・ドキュメント・タスク・データベースが全部1つに収まる万能ノート系の SaaS
  • 会社の Wiki・議事録・ナレッジ集の置き場として、Poolside 含め多くの組織で採用されている
  • 個人の無料プランで十分試せる。アカウントが無い人は https://notion.so で作成（会社の方針があれば居村に確認）

課題の流れ

Step 1：練習用フォルダを作って test-document.md を作る

Cursor で ~/dev/markdown-practice/ フォルダを開いて、内蔵ターミナルで Claude Code を起動してから、こう頼みます：

このフォルダに test-document.md を作ってください。
内容は「Markdown の主要な記号を一通り含む練習用ドキュメント」にしたいです。
具体的には以下を全部含めて：
・H1、H2、H3 の見出し（タイトル例：『私の Markdown 練習帳』）
・段落（短めの自己紹介を3〜5行）
・箇条書き（3項目）
・番号付きリスト（3項目）
・太字と斜体の例文
・インラインコード（例：command + V）
・コードブロック（HTML の Hello World 例を埋める）
・引用（誰かの言葉を1文引用）
・リンク（example.com に飛ばす）
・チェックリスト（- [ ] 未完了 と - [x] 完了）
機密情報は含めず、ダミーの内容で OK です。

Claude が一通り盛り込んだ Markdown ファイルを作ってくれます。Cursor で開いて、プレビューも確認してみてください。

Step 2：Markdown の中身を全選択してコピー

Cursor で test-document.md を開き、command + A で全選択 → command + C でコピー。

Cursor のプレビュー画面ではなく、記号が見えている状態（編集画面）からコピーしてください。 プレビュー画面からコピーすると、Notion 側で記号として認識されないことがあります。

Step 3：Notion で新しいページを作って貼り付け

  1. ブラウザで Notion を開き、サイドバーの「+ Add a page」または「+ ページを追加」
  2. 新しい空のページが開いたら、本文エリアにフォーカスしてそのまま command + V
  3. Notion が「これは Markdown ですね」と認識して自動で整形される

Step 4：整形のされ方を観察

貼り付けが終わったら、見た目を Markdown 元と見比べてみてください。

  • 見出し（#）→ 大きい文字に
  • 箇条書き（-）→ ドット付きリストに
  • 番号付き（1.）→ 番号付きリストに
  • 太字（**）→ ボールドに
  • コードブロック（バッククォート3つ）→ コードハイライト付きブロックに
  • チェックリスト（- [ ]）→ Notion のチェックボックスに
  • リンク（[text](url)）→ クリック可能なリンクに

完璧に変換されない要素もあります（Notion のバージョンや状況による）。 「Markdown は完璧な変換ではないが、9割以上自動で綺麗になる」感覚を掴むのが本課題の狙いです。

Step 5：AI に「次の運用」を聞いてみる

Markdown を Notion に貼り付けて整形する流れを試しました。
この運用が業務でどんな場面で活きるか、初心者向けに5パターン挙げて教えてください。
Poolside（映像制作・音声メディア・イベント・システム開発の会社）の文脈で、ありそうなシーンを想定して。

この課題で身につくこと

  • Markdown を「人間向けに展開する手段」があることを体験で理解できる
  • 議事録・原稿・ナレッジを .md で管理しつつ、共有時は Notion に流し込むワークフローが見える
  • AI に Markdown で生成させて、Notion に貼って完成形にする、というAI と人間の役割分担が掴める

Poolside での想定シーン

  • 議事録：Claude Code で議事録を .md 生成 → Notion に貼って関係者に共有
  • 提案書のドラフト：AI に Markdown で構成案を作らせて、Notion で整形・装飾
  • 社内ナレッジ：技術メモを .md で書いて GitHub に保存しつつ、Notion でも共有
  • ブログ下書き：Markdown で書いて、Notion 経由で関係者レビュー → CMS にコピペで投稿
Notion に貼り付ける時、機密情報が含まれていないか必ず再確認してください（第7章の方針）。 Notion のページは初期設定で「ワークスペース内に共有」されることがあるので、誤って機密情報を組織内に拡散しないよう注意。

Markdown を書いて Notion に流し込む運用を覚えると、生産性が一段上がる。 AI に Markdown で書かせる → Notion で見た目を整える → 共有、の流れを身につけろ。

逆に「Notion で最初から整形しながら書く」と思考のスピードが落ちる。 AI と組む時は、構造化されたテキスト（Markdown）で会話するのが圧倒的に速い。

うまく頼むコツ

  • 具体的に書く：「いい感じに」より「短くて読みやすい1段落で」のほうが伝わる
  • まず小さく頼む：いきなり「Web サイトを作って」より「index.html を作って中身は h1 タグだけ」
  • うまくいかなかった時は「いまのを取り消して」と頼む：戻すのも AI に頼める
  • 分からない言葉が出てきたら「これってどういう意味？」と素直に聞く：恥ずかしくない。むしろ最短ルート
さっき作ったファイルを、初心者向けに「どんなことをしたか」順番に解説してください。

コスト感覚を持つ — トークンと上手に付き合う

AI を使う上で必ず意識してほしいのが「トークン」という単位です。 AI は内部で文字を「トークン」という小さな塊に変換して処理しています。 だいたい 日本語1文字 ≒ 1〜2トークンと思ってください。

なぜコストの話をするのか

会社の業務でプランを使っている間は、月額固定で済むケースが多く、過剰に怖がる必要はありません。 ただし、API キー利用や Claude Code の本格運用では「使った分だけ課金」になることがあり、無自覚に大量にトークンを消費すると請求が膨らみます。 今のうちから節約のクセを付けておくと、将来本格運用に入る時にぐっと楽になります。

トークンを大きく消費する場面（要注意）

  • 巨大なファイルを丸ごと読ませる（数千行のコード、長い PDF など）
  • 同じ会話を何時間も続ける（履歴がどんどん膨らむ）
  • フォルダ全体を「全部読んで」と頼む（不要なファイルまで読み込む）
  • 長い出力を何度もやり直す（毎回同じくらいの量を再生成）

コストをかけずに使い倒す7つの習慣

  1. 用件は具体的に書く — 「この関数だけ直して」のように対象を絞ると、AI が読み込む範囲が狭くなりトークン節約になる
  2. 区切りで /clear する — 話題が一段落したら会話履歴をリセット。古い文脈を引きずらないことでトークン節約＋AI の精度も上がる
  3. /compact で履歴を要約 — 会話を続けたい時は /clear ではなく /compact で、過去のやり取りを要約してくれる
  4. 不要なファイルを .claude/settings.json や CLAUDE.md で除外 — node_modules、ビルド成果物などは AI に読ませない設定にする
  5. 長文を貼る前に「要約してから貼る」 — 100ページの議事録を全部貼るより、要約を貼る方が安く・的確に答えてもらえる
  6. 同じ作業はエージェント／スキルにする（第13章） — 毎回プロンプトを長く書くより、定型化して呼び出した方が安い
  7. ハマったら一度終了して新セッション — 解決しない時にダラダラ続けるより、別セッションで仕切り直した方が時間もトークンも節約になる
Claude Code の現在のコンテキスト使用量を教えてください。
また、今のセッションをこれ以上続けるべきか、いったん /clear して新しく始めるべきか、状況を見て助言してください。
有料プラン・API 従量課金への切替判断は、必ず居村に相談してください。 第1章の「お約束」にある通り、個人判断で課金プランを切り替えないこと。 無料枠で足りないと感じた段階で、まず相談を。

議事録の習慣 — 何をやったか必ず残す

Poolside では、Claude Code を使った作業のたびに 「手動ログ補完」議事録を残す運用になっています。 AI との会話履歴は claude が自動で保存しますが、「人間が後から読みやすい要約」を別途残すという文化です。

  • その日に何をやったか、どこまで進んだか
  • 決めたこと（決定事項）／決めなかったこと（未決事項）
  • 次のセッションで最初に取りかかること

この習慣の具体的なやり方・テンプレート・ファイル配置は、第13章の「議事録運用」で詳しく扱います。 本章ではまず「セッション後に手書きの議事録を残す」という習慣があることだけ、頭に入れておいてください。

練習フォルダの片付け

練習が一段落したら、Finder からゴミ箱にドラッグしても構いません。 あるいはこのまま残しておいて、別の練習に使ってもOK。
